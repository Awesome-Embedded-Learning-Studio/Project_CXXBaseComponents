# 00 - 源码阅读方法

## 阅读源码的正确姿势

拿到一个大型开源项目，第一反应往往是：这从何读起？代码文件成百上千，目录结构复杂，各种宏定义和模板混在一起，很容易让人望而生畏。

### 不要从 main 开始

很多人阅读源码的习惯是从 `main()` 函数开始，一行一行往下读。这对于小程序还行，对于大型项目简直是灾难。大型项目的 `main()` 往往很简短，真正的逻辑在各种初始化和回调里，跟踪起来会迷失方向。

### 自顶向下，先抓骨架

正确的阅读方式是：**先理解项目的宏观结构，再深入细节**。就像看一座建筑，先看平面图，再看每个房间的设计。

对于 C/C++ 项目，我建议的阅读顺序是：

1. **README 和文档**：了解项目是做什么的
2. **论文或设计文档**：理解核心设计思想
3. **目录结构**：了解代码组织方式
4. **头文件**：理解数据结构和接口
5. **核心实现**：选择一个核心功能深入

### 从 README 开始

首先看看项目的 README：

```bash
cd project/external/mimalloc
cat README.md
```

mimalloc 的 README 会告诉你：

- 这是一个微软开发的通用内存分配器
- 可以替代系统 malloc
- 有各种性能优化
- 支持多种平台

接下来看看目录结构：

```bash
ls -la
```

你会看到：
- `include/`：公共头文件
- `src/`：源代码
- `doc/`：文档
- `test/`：测试代码
- `CMakeLists.txt`：构建配置

### 阅读论文

mimalloc 有一篇配套论文，这是理解设计思想的关键。即使不完全读懂每个细节，也能了解：

- 为什么比系统 malloc 快
- 核心优化技术是什么
- 数据结构的设计思路

论文中提到的几个关键概念：

1. **Free List Shuffling**：打乱自由链表，提高缓存利用率
2. **Per-thread Heap**：每个线程独立的堆
3. **Partial Coalescing**：部分合并，减少碎片
4. **Size Classes**：大小分级管理

### 从数据结构入手

理解一个系统，最快的方式是理解它的数据结构。代码会变，但数据结构相对稳定。

mimalloc 的核心数据结构：

- `mi_heap_t`：线程本地的堆
- `mi_page_t`：内存页的管理结构
- `mi_block_t`：内存块的表示

我们从 `mi_heap_t` 开始，因为它是每个线程独立的入口点。

### 工具辅助

使用工具可以大大提高阅读效率：

1. **grep/ripgrep**：搜索函数和变量的定义
2. **ctags/cscope**：生成标签，快速跳转
3. **IDE**：VSCode/CLion 等的代码导航功能
4. **调试器**：gdb/lldb，动态观察运行状态

### 绘制架构图

### 带着问题阅读

不要被动地"读"代码，要主动地"问"代码：

- 这个结构体为什么要这样设计？
- 这个函数在什么情况下被调用？
- 这个锁保护的是什么？
- 这里的性能瓶颈在哪里？

带着具体的问题去读，效率会高很多。

### 先理解流程，再抠细节

第一次读代码，不要纠结每个宏、每个位操作。先理解大的流程：malloc 大致做什么，free 大致做什么。

等框架清晰了，再回头研究细节：某个位域是什么意思，某个宏展开后是什么。

### 对照测试代码

测试代码是最好的使用示例。看看 `test/` 目录下的测试：

```bash
ls test/
```

这些测试展示了 mimalloc 的各种使用场景，边看测试边读代码，更容易理解。

### 小结

阅读大型开源项目的正确姿势：

1. **先宏观后微观**：先理解架构，再抠细节
2. **从数据结构入手**：数据结构是系统的骨架
3. **带着问题阅读**：主动思考，不要被动接收
4. **善用工具**：grep、ctags、调试器都是好帮手
5. **画图总结**：把理解可视化
