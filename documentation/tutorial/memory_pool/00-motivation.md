# 00 - 为什么需要内存池

## malloc 哪里不行了

说实话，当我们第一次听到"内存池"这个词的时候，第一反应可能是：系统不是已经有 malloc 了吗？为什么还要自己折腾一套？

这个问题问得好。malloc 确实能用，而且用起来很方便。但在某些场景下，它的性能真的不够看。

### malloc 的性能瓶颈

让我们先理解 malloc 是怎么工作的。当你调用 `malloc(size)` 时，分配器需要：

1. 搜索空闲链表，找到足够大的块
2. 如果找不到，向系统申请更多内存（sbrk/mmap）
3. 可能需要分割块（如果找到的块比需要的大）
4. 更新元数据

当你调用 `free(ptr)` 时，分配器需要：

1. 根据 ptr 找到块的元数据
2. 标记块为空闲
3. 可能需要合并相邻的空闲块（减少碎片）
4. 可能需要决定是否把内存还给系统

这些操作看起来不复杂，但问题在于：

**第一，锁竞争**。标准 malloc 是线程安全的，内部有锁。在高并发场景下，多个线程同时分配/释放内存，锁竞争会变成瓶颈。你可能以为只有多线程程序才有这个问题，但即使是单线程程序，你用的库函数（比如 printf）内部可能也会分配内存，而这些库可能是多线程的。

**第二，内存碎片**。频繁分配和释放不同大小的块，会产生外部碎片——虽然总的空闲内存够用，但没有一块连续的足够大。malloc 试图通过合并空闲块来缓解这个问题，但这需要遍历空闲链表，时间复杂度是 O(n)。

**第三，缓存不友好**。malloc 分配的内存可能分散在堆的各个位置，访问时不连续，导致 CPU 缓存命中率低。

**第四，开销不小**。每个分配的块都有元数据（大小、标志位等），这通常占用 8-16 字节。对于频繁分配的小对象（比如 4 字节的结构体），这个开销比例非常大。

### 一个简单的测试

让我们写个简单的测试看看 malloc 的性能：

```cpp
#include <chrono>
#include <iostream>
#include <vector>

void test_malloc() {
    const int N = 1000000;
    std::vector<int*> ptrs(N);

    auto start = std::chrono::high_resolution_clock::now();

    for (int i = 0; i < N; ++i) {
        ptrs[i] = new int;
    }

    for (int i = 0; i < N; ++i) {
        delete ptrs[i];
    }

    auto end = std::chrono::high_resolution_clock::now();
    auto duration = std::chrono::duration_cast<std::chrono::microseconds>(end - start);

    std::cout << "malloc/free " << N << " times: " << duration.count() << " us\n";
}

int main() {
    test_malloc();
    return 0;
}
```

运行结果（具体数字因平台而异）：

```
malloc/free 1000000 times: 152347 us
```

150 多微秒，100 万次分配和释放。看起来还不错？但这是单线程的情况。如果是多线程并发分配，性能会明显下降。

### 内存池的思路

内存池的核心思想是：**提前分配一大块内存，然后从这块内存里"切"小块给用户**。

这样做有几个好处：

**第一，减少锁竞争**。每个线程可以有自己独立的缓存，大部分分配/释放操作不需要加锁。

**第二，减少碎片**。固定大小的分配（比如都是 16 字节）不会产生外部碎片。不同大小的分配可以用多个池分别管理。

**第三，缓存友好**。从同一块内存分配的对象在物理内存上可能相邻，提高缓存命中率。

**第四，降低开销**。不需要为每个小块存储元数据，元数据集中在池的管理结构里。

**第五，批量分配**。可以向系统申请大块内存，减少系统调用的次数。

### 什么时候用内存池

不是所有场景都适合用内存池。内存池适合：

1. **频繁分配/释放固定大小的对象**：比如网络服务器处理每个连接的缓冲区
2. **分配/释放非常频繁**：比如游戏引擎每帧分配大量临时对象
3. **多线程环境**：每个线程独立缓存，减少锁竞争
4. **对性能敏感**：比如高频交易系统

不适合用内存池的场景：

1. **分配大小变化很大**：需要很多个不同大小的池，增加复杂度
2. **分配不频繁**：内存池的初始化开销不值得
3. **对象生命周期不确定**：内存池通常需要批量归还

### 我们的内存池设计

在本教程中，我们要实现一个三层结构的内存池：

```
┌─────────────────────────────────────────────┐
│           Application Code                  │
└─────────────────────────────────────────────┘
                     ↓
┌─────────────────────────────────────────────┐
│            ThreadCache (thread_local)       │
│  每个线程独立的缓存，无锁分配/释放            │
└─────────────────────────────────────────────┘
                     ↓
┌─────────────────────────────────────────────┐
│            CentralPool (全局共享)            │
│  管理多个 Size Class，线程安全                │
└─────────────────────────────────────────────┘
                     ↓
┌─────────────────────────────────────────────┐
│               System Memory                 │
│        ::operator new / delete              │
└─────────────────────────────────────────────┘
```

- **ThreadCache**：每个线程独立，无锁快速分配
- **CentralPool**：所有线程共享，管理空闲内存
- **Size Classes**：将大小划分为固定的档位（8/16/24/.../128 字节）

### Size Classes 设计

我们将内存大小划分为多个档位：

```cpp
static constexpr size_t kMaxSmallSize = 128;  // 最大支持 128 字节
static constexpr size_t kClassGrid = 16;      // 每 16 字节一档
```

这样就有 128/16 = 8 个档位：

| 档位索引 | 大小范围 | 实际分配大小 |
|---------|----------|-------------|
| 0 | 1-16 | 16 |
| 1 | 17-32 | 32 |
| 2 | 33-48 | 48 |
| ... | ... | ... |
| 7 | 113-128 | 128 |

每个档位对应一个 FreeList，只分配固定大小的块。这样不会产生外部碎片。

### 小结

到这里，我们理解了为什么需要内存池，以及我们的内存池将采用什么设计。malloc 虽然通用，但在高频分配/释放场景下性能不够理想。内存池通过提前分配、线程本地缓存、Size Classes 等技术，可以显著提升性能。
