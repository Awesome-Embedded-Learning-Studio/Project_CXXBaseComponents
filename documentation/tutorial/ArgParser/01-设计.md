# 01 - 数据结构先行：如何表示一个命令行参数

## 一个参数需要记录什么

现在我们进入正题，开始设计 ArgParser 的数据结构。在做任何实现之前，先想清楚"需要表示什么"往往比"怎么实现"更重要。那么问题来了——一个命令行参数，到底需要记录哪些信息呢？

让我们以 `--output file.txt` 这个参数为例来思考。首先，用户需要通过某种方式来识别这个参数，所以我们需要一个短名 `-o` 和一个长名 `--output`，这两种形式都能指向同一个选项。然后，当用户输入 `--help` 的时候，我们希望能打印一段帮助文字来说明这个参数的作用，所以需要一个 `help` 字段。

接下来是更重要的逻辑属性。这个参数是必填的还是可选的？如果是必填的但用户没有提供，我们需要在解析后报错提示。这个参数是一个布尔开关（比如 `-v` 表示 verbose），还是需要带值的参数（比如 `--output file.txt`）？这两种参数的处理逻辑完全不同，开关类参数只要出现就表示 true，而带值参数还要继续读取下一个 token 作为它的值。

然后是存储解析结果的部分。我们需要一个字段来存储参数的值，这个值应该用什么类型呢？考虑到命令行输入本质上都是字符串，不管是数字、路径还是其他什么类型，最初都是以字符串的形式存在的。所以我们用 `std::string` 来存储，等到用户取值的时候再根据需要转换成对应的类型。这里还有一个容易被忽略的细节——如何区分"用户没有传这个参数"和"用户传了一个空字符串"？这两种情况语义完全不同，所以我们还需要一个 `is_set` 布尔字段来标记这个参数是否被设置过。

把这些想法整理一下，我们就能得到 `Arg` 结构体的定义：

```cpp
struct Arg {
    std::string short_name;  // "-o"
    std::string long_name;   // "--output"
    std::string value;       // 存储的值（字符串形式）
    std::string help;        // 帮助文字
    bool required{false};    // 是否必填
    bool is_flag{false};     // 是否是布尔开关
    bool is_set{false};      // 是否已被设置
};
```

## 用什么容器存储所有参数

定义好单个参数的结构之后，接下来要考虑的是用什么容器来存储所有的参数。这里有几个选择——可以用 `std::vector` 按注册顺序存储，也可以用 `std::map` 或 `std::unordered_map` 按键值对的方式存储。

我们选择 `std::unordered_map<std::string, Arg>`，key 是长名去掉 `--` 之后的部分。也就是说，如果用户注册了 `--output` 参数，那么它在 map 中的 key 就是 `"output"`。这样做的好处是，当用户调用 `get("output")` 来取值时，我们可以用 O(1) 的时间复杂度直接定位到对应的参数，而不需要遍历整个容器。

你可能会问，那短参数怎么办？比如用户输入的是 `-o`，我们怎么找到对应的 `Arg`？这个问题确实存在，但它的解决方式相对简单——短参数的查找频率通常不会太高，我们可以用一个线性遍历的辅助函数来实现，虽然时间复杂度是 O(n)，但考虑到参数数量通常不会很多，这个开销完全可以接受。

```cpp
Arg* findLongByShort(const std::string& short_name) {
    for (auto& [key, arg] : args_) {
        if (arg.short_name == short_name) {
            return &arg;
        }
    }
    return nullptr;
}
```

## 位置参数需要单独处理

现在我们来考虑位置参数。位置参数和命名参数有一个本质的区别——它们没有 `--` 前缀，是按照在命令行中出现的顺序来匹配的。比如 `cp src.txt dst.txt`，第一个裸值对应源文件，第二个裸值对应目标文件。

这意味着位置参数的定义和命名参数应该分开存储。我们需要一个结构体来描述位置参数的元信息：

```cpp
struct PosArg {
    std::string name;   // 参数名，比如 "src" 或 "dst"
    std::string help;   // 帮助文字
    bool required;      // 是否必填
};
```

然后用一个 `std::vector<PosArg>` 来存储所有位置参数的定义。为什么要用 vector 而不是 map？因为位置参数是按顺序匹配的，注册的顺序就是匹配的顺序，vector 能完美保持这个顺序。

解析的时候，我们先收集所有的裸值，然后按照注册顺序依次分配给位置参数。如果某个位置参数是必填的但没有对应的裸值，就抛出异常。解析后的值存放在另一个 map 里，key 是参数名，value 是解析到的字符串值：

```cpp
std::vector<PosArg> pos_args;                        // 位置参数定义
std::unordered_map<std::string, std::string> pos_values;  // 位置参数值
```

## 内部辅助函数

接下来我们需要一个辅助函数，把长名参数转换成 map 的 key。这个逻辑很简单，就是把前两个字符 `--` 去掉：

```cpp
namespace {
std::string longNameToKey(const std::string& long_name) {
    return long_name.substr(2);  // "--output" -> "output"
}
}
```

把它放在匿名命名空间里，这样它只在当前编译单元内可见，不会污染全局命名空间。这个函数会在很多地方用到——注册参数的时候、解析参数的时候，都需要把 `--output` 转换成 `"output"` 来作为 key。

## 完整的类成员变量

把上面所有的内容整合起来，ArgParser 类的私有成员变量大概就是这样的：

```cpp
class ArgParser {
private:
    std::unordered_map<std::string, Arg> args_;      // 命名参数存储
    std::vector<PosArg> pos_args;                    // 位置参数定义
    std::unordered_map<std::string, std::string> pos_values;  // 位置参数值

    Arg* findLongByShort(const std::string& short_name);
};
```

这里有一个设计细节值得注意——我们把 `args_` 设计成 `unordered_map` 而不是 `map`，主要是出于性能考虑。参数的查找操作会在解析过程中频繁发生，unordered_map 的 O(1) 平均查找时间比 map 的 O(log n) 更有优势。当然，unordered_map 的缺点是元素遍历的顺序是不确定的，但在这个场景下，我们可以在打印帮助信息的时候手动排序，所以这不是一个致命问题。

到这里，数据结构的设计就基本完成了。你会发现，我们用两个结构体、三个容器，就完整地表示了一个命令行解析器需要维护的所有信息。接下来要做的事情，就是提供接口让用户能够注册参数，然后实现解析逻辑把这些容器填满。

下一步，我们来实现注册接口——也就是让用户能够告诉 ArgParser，他们的程序需要哪些参数。
