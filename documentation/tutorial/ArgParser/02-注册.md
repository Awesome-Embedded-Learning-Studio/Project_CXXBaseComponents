# 02 - 注册接口：让用户描述参数

## 三种注册函数

数据结构设计好了，现在我们提供接口让用户能够注册参数。根据我们在上一篇文章中分析的参数类型，我们需要三个注册函数：`add_argument` 用于带值的命名参数，`add_flag` 用于布尔开关，`add_positional` 用于位置参数。

先来看 `add_argument`。这个函数接收短名、长名、帮助文字，以及一个可选的 `required` 参数，默认为 false。函数体内部做的事情很简单——把长名转换成 key，然后用指定的值构造一个 `Arg` 对象并存入 map：

```cpp
void add_argument(const std::string& short_name,
                  const std::string& long_name,
                  const std::string& help,
                  bool required = false) {
    const std::string key = longNameToKey(long_name);
    args_[key] = Arg{
        .short_name = short_name,
        .long_name = long_name,
        .value = "",
        .help = help,
        .required = required,
        .is_flag = false,  // 这不是开关，是带值的参数
        .is_set = false
    };
}
```

这里用到了 C++20 的指定初始化器语法，可以直接用 `.` 后跟字段名的方式来初始化结构体，可读性比传统的大括号初始化要好很多。注意这里的 `is_flag` 被设置为 false，这是我们区分带值参数和布尔开关的关键标记。

## 布尔开关的特殊处理

接下来是 `add_flag`。布尔开关是一种特殊的命名参数，它们的特点是不需要值——只要出现在命令行中，就表示这个选项被启用了。典型的例子就是 `-v` 或 `--verbose` 这种选项。

```cpp
void add_flag(const std::string& short_name,
              const std::string& long_name,
              const std::string& help) {
    const std::string key = longNameToKey(long_name);
    args_[key] = Arg{
        .short_name = short_name,
        .long_name = long_name,
        .value = "",
        .help = help,
        .required = false,  // 开关永远不可能是必填的
        .is_flag = true,    // 标记这是一个开关
        .is_set = false
    };
}
```

你会发现这个函数和 `add_argument` 非常相似，唯一的区别在于 `is_flag` 被设置为 true，而且 `required` 固定为 false。这符合直觉——一个开关要么存在要么不存在，不存在也不应该算作错误，所以它永远不可能是必填的。

你可能会问，为什么不设计成一个通用的 `add_argument` 函数，然后用一个参数来区分是不是 flag 呢？确实可以这么做，但我觉得从 API 的语义上来说，分开两个函数会更清晰。用户在写代码的时候，看到 `add_flag` 就能立刻明白这是注册一个开关选项，而不需要再去查文档或者看函数签名来确认那个 bool 参数到底代表什么。

## 位置参数的注册

位置参数的注册相对简单一些。因为位置参数没有短名和长名，只有一个名字用于标识，所以我们只需要传入名字、帮助文字，以及是否必填：

```cpp
void add_positional(const std::string& name,
                    const std::string& help,
                    bool required = true) {
    pos_args.emplace_back(PosArg{
        .name = name,
        .help = help,
        .required = required
    });
}
```

这里注意我们把 `required` 的默认值设为了 true，和命名参数相反。这是因为位置参数通常更像是"核心输入"，比如输入文件路径这种东西，大多数情况下都是必填的。当然，用户也可以显式传入 false 来表示这个位置参数是可选的。

另外这里用的是 `emplace_back` 而不是 `push_back`。`emplace_back` 会直接在 vector 的末尾构造对象，而 `push_back` 会先构造一个临时对象再拷贝进去。对于像 `PosArg` 这样的简单结构体，两者的性能差异几乎可以忽略，但 `emplace_back` 的语义更准确——我们是"在容器中放置一个新元素"而不是"把一个已有元素放进去"。

## 注册顺序很重要

位置参数是按照注册顺序来匹配的，这一点非常关键。比如你先注册了 `src`，再注册了 `dst`，那么命令行的第一个裸值就会被分配给 `src`，第二个裸值分配给 `dst`。如果顺序搞反了，整个语义就乱了。

```cpp
// 正确的注册顺序
parser.add_positional("src", "源文件路径");
parser.add_positional("dst", "目标文件路径");

// 命令行: ./program input.txt output.txt
// src -> "input.txt"
// dst -> "output.txt"
```

所以在写代码的时候，一定要确保位置参数的注册顺序和你的预期匹配。这一点在文档里必须写清楚，不然用户很容易踩坑。

## 命名参数的顺序无关

和位置参数不同，命名参数的注册顺序完全不影响解析结果。因为命名参数是通过 `--key` 的形式来识别的，不管你在代码里先注册 `--output` 还是先注册 `--input`，解析的时候都是根据命令行中实际出现的参数名来匹配的。

这一点对用户很友好——你可以在代码里按照逻辑分组来注册参数，而不需要担心顺序问题。比如你可以把所有输入相关的参数放在一起，所有输出相关的参数放在一起，这样代码的可读性会更好。

## 容器的选择与设计

到这里你可能已经注意到了，我们用 `unordered_map` 来存储命名参数，用 `vector` 来存储位置参数的定义。这两种容器的选择是有原因的，反映了两种参数的不同特性。

命名参数需要通过名字来查找，而且查找操作会在解析过程中频繁发生，所以用哈希表能提供 O(1) 的平均查找时间。位置参数则是按顺序匹配的，我们需要保持注册顺序，所以用 vector 更合适。

解析后的位置参数值是用另一个 `unordered_map` 来存储的，key 是参数名，value 是解析到的字符串值。这样做的好处是，不管用户想查询的是命名参数还是位置参数，都可以用同一个 `get(key)` 函数，内部会先查命名参数的 map，再查位置参数的 map。这种设计对用户来说是透明的，他们不需要关心参数是哪一种类型，只要知道参数名就能拿到值。

## 注册完成之后

当用户把所有需要的参数都注册完之后，`args_` 和 `pos_args` 这两个容器就被填满了。这时候我们有了完整的参数定义，知道了每个参数的名字、帮助文字、是否必填、是不是开关等等信息。下一步就是解析命令行输入，把用户实际传入的值填进去。

接下来我们要面对的是整个 ArgParser 最复杂的部分——`parse()` 函数。这个函数需要逐个扫描命令行 token，根据不同的格式做不同的处理，同时还要处理各种错误情况。这是核心中的核心，也是最容易出 bug 的地方。
