# 00 - 从零开始：为什么我们需要一个命令行解析器

## 事情的起因

老实说，这个想法的诞生纯粹是因为我实在受够了在 main 函数里写一堆 if-else 去处理命令行参数。你肯定也有过类似的经历——一开始只是想写个简单的小工具，比如一个图片转换程序，第一版代码可能长这样：

```cpp
int main(int argc, char* argv[]) {
    std::string input = argv[1];
    std::string output = argv[2];
    // 做一些处理...
}
```

用起来倒是挺直接，`./mytool input.jpg output.png`，也没什么问题。但好景不长，用户的需求总是会像滚雪球一样越滚越多。有人问你能不能支持自定义压缩质量，有人想要一个 `-v` 参数来看详细日志，还有人说参数顺序老是记不住，能不能写成 `--input input.jpg` 这种形式。

于是你开始在 main 里到处补丁，代码很快就变成了这个样子：

```cpp
for (int i = 1; i < argc; i++) {
    if (strcmp(argv[i], "--verbose") == 0 || strcmp(argv[i], "-v") == 0) {
        verbose = true;
    } else if (strcmp(argv[i], "--input") == 0) {
        input = argv[++i];
    } else if (strcmp(argv[i], "--output") == 0) {
        output = argv[++i];
    }
    // 更多 else if...
}
```

说实话，写到这个时候我就已经开始怀疑人生了。每加一个参数就要复制粘贴一堆 if-else，还要处理短名、长名、带值的、不带值的，各种边界条件一想起来就让人头大。更糟糕的是，如果用户填错了参数，你还得给出一个友好的报错提示，不然他们就会给你发各种莫名其妙的 bug 报告。

## 一个通用的解决方案

这时候你会意识到，命令行参数解析实际上是一个相当通用的编程问题。不管是写图像处理工具、编译器，还是服务器程序，大家都要面对同样的需求：支持短参数和长参数，处理位置参数，提供帮助信息，做参数校验。既然这是一个通用问题，那就值得把它抽象成一个专门的类来解决，而不是每次都重新造轮子。

我们想要的最终效果大概是这样的——用一个简洁的 API 来描述你的参数，然后让解析器帮你处理所有的脏活累活：

```cpp
ArgParser parser;

parser.add_positional("src", "源文件路径");
parser.add_argument("-o", "--output", "输出文件路径");
parser.add_flag("-v", "--verbose", "启用详细输出");

parser.parse(argc, argv);

std::cout << parser.get<std::string>("src") << "\n";
std::cout << parser.get_flag("verbose") << "\n";
```

这样一来，main 函数就能保持干净，参数定义和业务逻辑清晰分离。更重要的是，所有的解析逻辑、错误处理、帮助信息生成都被封装在了 ArgParser 内部，你只需要关注"我需要哪些参数"这个问题本身。

## 为什么不直接用现成的库

你可能会问，市面上不是有现成的命令行解析库吗，比如 getopt、Boost.Program_options、CLI11 之类的。确实，这些库都很成熟，功能也很强大。但问题在于，它们要么是 C 风格的 API 写起来很啰嗦，要么依赖了一大堆其他库，编译配置比写代码还累。

而且，从学习的角度来看，亲手实现一个命令行解析器是一个相当不错的练习。它会让你深入理解 C++ 的模板编程、字符串处理、STL 容器的使用，甚至涉及到一些 API 设计的思考。当你把这个东西从头到尾写一遍之后，再去看那些成熟的库，你会发现它们的很多设计决策其实都是有迹可循的。

## 我们的目标

好，废话不多说，让我们明确一下要实现的目标。我们的 ArgParser 需要支持以下功能：

首先是三种基本的参数类型。位置参数是那些不带前缀、按顺序匹配的值，比如 `cp file1 file2` 中的 file1 和 file2。命名参数则是带有 `--` 或 `-` 前缀的参数，比如 `--output=file.txt` 或者 `-o file.txt`。布尔开关是一种特殊的命名参数，它们不需要值，出现就表示 true，典型的例子就是 `-v` 或 `--verbose` 这种开关选项。

然后是两种主要的参数格式。一种是 `--key=value` 这种等号连接的写法，另一种是 `--key value` 这种空格分开的写法，我们的解析器需要同时支持这两种形式。短参数和长参数需要能够互通，比如 `-v` 和 `--verbose` 应该指向同一个选项。

最后，我们需要一个 `--help` 功能，能够自动生成格式化的帮助信息，让用户一眼就能看出这个程序支持哪些参数、哪些是必填的、每个参数的作用是什么。当用户输入了未知的参数或者缺少必填参数时，解析器应该给出清晰的错误提示，而不是直接崩溃或者产生莫名其妙的行为。

## 接下来怎么做

现在我们已经明确了需求，接下来就要开始动手实现了。整个过程我们会采用迭代式的开发方式，从最简单的数据结构开始，逐步添加功能，每一步都会讲清楚为什么要这样设计。你会发现，一个看似复杂的命令行解析器，其实可以被拆解成几个相对独立的小问题：如何表示一个参数、如何注册参数、如何解析命令行输入、如何读取解析结果、如何生成帮助信息。

让我们先从最基础的问题开始——一个命令行参数，到底需要记录哪些信息？
