# 03 - 解析核心：读懂命令行输入

## parse() 函数的职责

现在我们来到整个 ArgParser 最核心的部分——`parse()` 函数。这个函数接收 `argc` 和 `argv`，也就是命令行参数的数量和数组，然后逐个扫描这些 token，把它们识别成不同类型的参数并解析出对应的值。

这个过程听起来简单，但实际做起来会发现有很多细节需要处理。我们来逐步拆解这个函数的逻辑，看看它是如何处理各种不同格式的命令行输入的。

## Token 的四种类型

首先我们需要对命令行中的每个 token 进行分类。一个典型的命令行可能是这样的：

```
./mytool src.txt --count=10 -v dst.txt
```

从 `argv[1]` 开始（`argv[0]` 是程序名），我们会遇到四种不同类型的 token：

第一种是 `--help` 或 `-h`，这是帮助请求，遇到就直接打印帮助信息并退出程序。第二种是 `--key=value` 格式，长名参数带等号连接值。第三种是 `--key` 或 `-k` 格式，需要从下一个 token 读取值。第四种是裸值，也就是不属于上述任何类型的字符串，它们会被收集起来作为位置参数的候选值。

## 处理帮助请求

我们先来看最简单的逻辑——处理 `--help` 或 `-h`。这个检查需要在循环的最开始就做，因为一旦用户请求帮助，就没有必要继续解析了：

```cpp
for (int i = 1; i < argc; i++) {
    std::string_view token{argv[i]};

    if (token == "--help" || token == "-h") {
        print_help(argv[0]);
        std::exit(0);
    }
    // ...
}
```

这里有一个细节值得注意——我们用的是 `||` 而不是 `&&`。这是一个非常容易犯的错误，我一开始写的时候差点就写成 `&&` 了，那样的话这个条件永远都不会成立（一个 token 不可能同时等于 `--help` 和 `-h`）。用 `||` 才是正确的逻辑——只要 token 等于其中任意一个，就打印帮助。

另外我们用的是 `std::string_view` 而不是 `std::string`。`string_view` 是 C++17 引入的类型，它只是一个对字符串的"非拥有性引用"，不需要分配内存或拷贝数据。在这个场景下非常适合，因为我们只是需要比较 token 的内容，不需要修改它或持有它。

## 处理 --key=value 格式

接下来是处理 `--key=value` 格式的长名参数。这种格式的特点是参数名和值在同一个 token 里，用等号连接：

```cpp
if (token.size() > 2 && token[0] == '-' && token[1] == '-') {
    auto equal_pos = token.find('=');
    if (equal_pos != std::string::npos) {
        // 提取 key: 从第2个字符开始，长度为 equal_pos-2
        std::string_view key_view = token.substr(2, equal_pos - 2);
        std::string key{key_view};

        auto it = args_.find(key);
        if (it == args_.end()) {
            throw std::invalid_argument("Unknown argument: --" + key);
        }

        // 提取 value: 从等号后一个字符开始到结尾
        it->second.value = token.substr(equal_pos + 1);
        it->second.is_set = true;
    }
    // ...
}
```

这里有几个关键点需要解释。首先我们检查 token 的长度大于 2 且前两个字符都是 `-`，这能确保我们处理的是长名参数而不是单个 `-`。然后用 `find('=')` 来查找等号的位置，如果找到了就说明这是 `--key=value` 格式。

提取 key 的时候用的是 `substr(2, equal_pos - 2)`，这里的参数需要仔细理解一下。`substr(pos, len)` 的第一个参数是起始位置，第二个参数是长度。我们从第 2 个字符开始（跳过 `--`），长度是 `equal_pos - 2`，也就是从开头到等号之前但不包括等号的部分。比如 `--count=10`，`equal_pos` 是 7，那么 `substr(2, 5)` 得到的就是 `"count"`。

提取 value 的时候用的是 `substr(equal_pos + 1)`，这里只传了一个参数，表示从等号后一个字符开始到字符串结尾。如果 `--count=10` 的 `equal_pos` 是 7，那么 `substr(8)` 得到的就是 `"10"`。

最后我们把解析到的 key 在 `args_` map 中查找，如果找不到就说明用户输入了一个未注册的参数，抛出异常。找到了就设置对应的值，并把 `is_set` 标记为 true。

## 处理 --key value 格式

如果长名参数后面没有等号，那它可能是 `--key value` 格式，也就是参数名和值分别在两个相邻的 token 里：

```cpp
else {
    std::string_view key_view = token.substr(2);
    std::string key{key_view};

    auto it = args_.find(key);
    if (it == args_.end()) {
        throw std::invalid_argument("Unknown argument: --" + key);
    }

    if (it->second.is_flag) {
        // 这是一个开关，不需要值
        it->second.is_set = true;
    } else if (i + 1 < argc) {
        // 这是一个带值参数，消费下一个 token 作为值
        it->second.value = argv[++i];
        it->second.is_set = true;
    } else {
        throw std::invalid_argument("Missing value: " + std::string{token});
    }
}
```

这里的逻辑稍微复杂一点。首先提取 key，这部分和之前一样，只是这次没有等号，所以直接 `substr(2)` 就能得到完整的参数名。

然后在 map 中查找对应的参数。如果找到了，需要判断它是不是一个 flag。如果是 flag，就直接把 `is_set` 设为 true，不需要读取值。如果不是 flag，那就需要从下一个 token 读取值。

这里有一个非常关键的细节——`argv[++i]`。我们在循环里用到了 `++i` 来提前消费下一个 token。这是因为当前 token 是参数名，下一个 token 是它的值，当我们在当前迭代中把值读进来之后，下一次循环就应该跳过这个值 token，不然它会被误认为是位置参数。这里的 `++i` 是前缀递增，会先增加 i 的值再使用，正好符合我们的需求。

当然，在读取下一个 token 之前，我们必须先检查 `i + 1 < argc`，确保确实有下一个 token。如果用户只写了 `--output` 而没有写值，我们就需要抛出一个友好的错误提示。

## 处理短参数

短参数的处理逻辑和长参数非常相似，唯一的区别在于我们需要用短名来查找对应的 Arg：

```cpp
if (token.size() >= 2 && token[0] == '-') {
    std::string short_key = std::string{token};
    Arg* arg = findLongByShort(short_key);

    if (!arg) {
        throw std::invalid_argument("Unknown argument: " + short_key);
    }

    if (arg->is_flag) {
        arg->is_set = true;
    } else if (i + 1 < argc) {
        arg->value = argv[++i];
        arg->is_set = true;
    } else {
        throw std::invalid_argument("Missing value: " + short_key);
    }
    continue;
}
```

这里我们调用了之前定义的 `findLongByShort` 函数来通过短名查找 Arg。找到之后的逻辑就和长参数完全一样了——如果是 flag 就标记为已设置，否则读取下一个 token 作为值。

## 收集裸值

所有不属于上述类型的 token 都会被认为是裸值，也就是位置参数的候选值：

```cpp
pos_arg_view.emplace_back(token);
```

我们把这些裸值收集到一个 vector 里，等循环结束之后再按注册顺序分配给位置参数。

## 分配位置参数

循环结束后，我们需要把收集到的裸值分配给位置参数：

```cpp
for (size_t i = 0; i < pos_args.size(); ++i) {
    if (i < pos_arg_view.size()) {
        // 有对应的裸值，分配给这个位置参数
        pos_values[pos_args[i].name] = pos_arg_view[i];
    } else if (pos_args[i].required) {
        // 没有对应的裸值，但这个参数是必填的
        throw std::invalid_argument("Required position args: <" +
                                    pos_args[i].name + ">");
    }
    // 如果不是必填的且没有提供值，就不做任何处理
    // get() 的时候会抛出异常
}
```

这里的逻辑是按注册顺序遍历所有位置参数定义，如果有对应的裸值就分配给它。如果裸值不够了，就需要检查当前这个位置参数是不是必填的，如果是就抛出异常。如果不是必填的，就什么都不做，等用户调用 `get()` 的时候再报错。

## 检查必填命名参数

最后一步是检查所有必填的命名参数是否都被设置了：

```cpp
for (auto& [key, arg] : args_) {
    if (arg.required && !arg.is_set) {
        throw std::invalid_argument("Required args: " + key);
    }
}
```

这里遍历所有命名参数，如果一个参数被标记为必填但 `is_set` 是 false，就抛出异常。

到这里，整个 `parse()` 函数的逻辑就完整了。你会发现，虽然代码看起来有点长，但其实每一段逻辑都很清晰——识别 token 类型，做对应的处理，然后继续下一个 token。关键是把各种边界情况都想清楚，比如等号格式、空格格式、短名、长名、flag、带值参数、位置参数、必填检查，这些都是缺一不可的。

解析完成之后，所有的参数值都被存储在 `args_` 和 `pos_values` 这两个容器里。下一步就是提供接口让用户能够读取这些值。这就是我们下一篇文章要讲的内容——`get<T>()` 模板函数的设计与实现。
