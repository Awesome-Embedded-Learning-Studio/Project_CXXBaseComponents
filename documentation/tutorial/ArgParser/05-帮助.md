# 05 - 用户体验：实现优雅的 --help

## 帮助信息的重要性

说实话，一个命令行工具的帮助信息做得好不好，直接影响用户的第一印象。你肯定有过这样的经历——想用一个新的工具，输入 `--help` 想看看支持什么参数，结果要么输出格式乱七八糟，要么信息不全，要么干脆就没有帮助功能。这种时候真的会让人血压拉满。

一个好的帮助信息应该做到几件事：让用户一眼就能看出程序的基本用法、列出所有支持的参数、说明每个参数的作用、标注哪些是必填的、格式整齐美观。这些看起来是细节，但积累起来就能决定用户愿不愿意继续用你的工具。

## print_help() 的整体结构

我们的 `print_help()` 函数需要打印三个部分的内容——第一行是程序的基本用法，然后是位置参数列表，最后是命名参数列表。对于每一部分，我们需要合理地安排格式，让信息层次清晰、对齐整齐。

```cpp
void ArgParser::print_help(std::string_view program_name) const {
    // 打印基本用法
    std::cout << "Usage: " << program_name;

    for (const auto& pd : pos_args) {
        std::cout << " <" << pd.name << ">";
    }
    std::cout << " [options]\n\n";
```

第一行是 "Usage: " 开头，后面跟着程序名和所有位置参数的占位符。每个位置参数用尖括号括起来，比如 `<input>`、`<output>`，这是一种常见的约定，表示这里需要一个值。最后加上 `[options]` 表示还有可选的命名参数。

这里 `program_name` 参数是 `argv[0]`，也就是用户调用程序时用的名字。这样做的好处是帮助信息会自动适配，不管用户把可执行文件叫什么名字，帮助信息都能正确显示。

## 位置参数列表

接下来是位置参数列表。如果程序有位置参数，我们就先打印一个标题 "Positional arguments:"，然后逐行列出每个位置参数的信息：

```cpp
if (!pos_args.empty()) {
    std::cout << "Positional arguments:\n";
    for (const auto& pd : pos_args) {
        std::string label = "<" + pd.name + ">";
        std::cout << "  " << std::left << std::setw(24) << label << pd.help;
        if (!pd.required) {
            std::cout << " [optional]";
        }
        std::cout << "\n";
    }
    std::cout << "\n";
}
```

这里的格式化输出用到了 `<iomanip>` 头文件里的 `std::setw` 和 `std::left`。`std::setw(24)` 设置下一个输出的宽度为 24 个字符，`std::left` 表示左对齐。这样参数名这一列就固定占 24 个字符，后面的帮助文字会自然地从同一列开始，整体看起来非常整齐。

你可能会有疑问，为什么是 24 这个数字？其实这是一个经验值，经过测试发现大多数参数名都能在 24 个字符内放得下，而且打印出来的效果也比较美观。如果你的程序有特别长的参数名，可以适当调整这个数字。

另外注意我们在可选的位置参数后面加上了 `[optional]` 标记。这样用户一眼就能看出哪些参数是必须传的，哪些可以省略。

## 命名参数列表

位置参数列表打印完之后，接下来是命名参数列表。逻辑类似，但格式稍有不同：

```cpp
std::cout << "Options:\n";
for (const auto& [key, arg] : args_) {
    std::string flags = arg.short_name + ", " + arg.long_name;
    if (!arg.is_flag) {
        flags += " <value>";
    }
    std::cout << "  " << std::left << std::setw(24) << flags << arg.help;
    if (arg.required) {
        std::cout << " [required]";
    }
    std::cout << "\n";
}
```

这里我们把短名和长名合并在一起显示，中间用逗号和空格隔开，比如 `-h, --help`。这是一种常见的格式，用户一眼就能看出这个参数有两种写法。

如果这个参数不是 flag，我们就在后面加上 ` <value>` 占位符，提醒用户这里需要跟一个值。如果是 flag，就不加任何占位符，因为 flag 本身不需要值。

必填参数后面会加上 `[required]` 标记，和位置参数的处理逻辑一样，只是方向相反——位置参数默认是必填的，可选的需要标注；命名参数默认是可选的，必填的需要标注。

## 帮助选项本身

最后别忘了，`-h` 和 `--help` 这个帮助选项本身也需要出现在列表里。虽然我们没有把它注册到 `args_` 里，但用户需要知道这个功能是存在的：

```cpp
std::cout << "  " << std::left << std::setw(24) << "-h, --help"
          << "Show this help message\n";
```

这样用户在打印的帮助信息里就能看到 `-h, --help` 这一项，知道可以用这个选项来查看帮助。

## 完整的输出效果

把所有代码组合起来，假设我们的 demo 程序注册了一些参数，打印出来的帮助信息大概是这样的：

```
Usage: ./demo <input> [options]

Positional arguments:
  <input>                输入文件路径

Options:
  -o, --output <value>   输出文件路径 [required]
  -n, --count <value>    处理数量（默认 1）
  -v, --verbose          启用详细输出模式
  -f, --force            强制覆盖已存在文件
  -h, --help             Show this help message
```

你会发现，所有的参数名、占位符、帮助文字都整齐地对齐在同一个列上，看起来非常专业。这种格式化的输出方式比简单的 `cout << help << endl` 要美观得多，也更容易阅读。

## 格式化的代价

使用 `std::setw` 和 `std::left` 进行格式化输出虽然效果好，但确实有一定的代码复杂度。你需要计算合适的列宽，处理不同长度的字符串，还要确保特殊字符（比如尖括号、方括号）能正确显示。

但从用户体验的角度来说，这些额外的工作是完全值得的。一个混乱的帮助信息会让用户产生"这个工具不专业"的第一印象，而一个整齐美观的帮助信息则能传递出作者的用心和专业性。

## 何时打印帮助

帮助信息的打印时机有两种。一种是用户主动请求，比如输入了 `--help` 或 `-h`，这时我们打印帮助并正常退出。另一种是参数解析出错，比如输入了未知的参数或者缺少必填参数，这时我们在打印错误信息之后，也会提示用户可以用 `--help` 查看完整的帮助信息。

```cpp
try {
    parser.parse(argc, argv);
} catch (const std::exception& e) {
    std::cerr << "[错误] " << e.what() << "\n";
    std::cerr << "使用 --help 查看帮助信息\n";
    return 1;
}
```

这样用户在遇到错误的时候，就知道可以通过 `--help` 来了解正确的用法。

到这里，ArgParser 的核心功能就都实现了——数据结构设计、参数注册、命令行解析、值读取、帮助信息打印。最后一篇文章，我们会用一个完整的示例程序来走通整个流程，看看这些组件是如何协同工作的。
