# 00 - 前言与动机

## 为什么我们又要造轮子

说实话，当我第一次决定手写一个命令行参数解析器的时候，心里也是犯嘀咕的。毕竟市面上现成的方案一抓一大把：POSIX 的 `getopt`、GNU 的 `getopt_long`、Boost.Program_options、还有一堆第三方库。放着现成的不用，非要自己折腾，这不是典型的重复造轮子吗？

但事情没那么简单。让我们先看看这些"现成方案"到底有什么问题。

### 现有方案的困境

`getopt` 这个老家伙是上世纪七十年代的东西了，虽然它能工作，但用起来真的让人血压拉高。你得手动维护一长串的 option string，比如 `"ab:c::"` 这种天书般的语法，稍微写错一个字符你就等着调试吧。而且它是纯 C 的 API，在现代 C++ 里用起来就像在宝马车里挂了个手动挡——能跑，但别扭。

Boost.Program_options 倒是现代多了，但这个依赖重量级得吓人。为了解析几个命令行参数引入整个 Boost，这感觉就像为了煎个蛋盖了一座厨房。而且它的编译时间、链接复杂度，对于小项目来说简直是杀鸡用牛刀。

更关键的是，无论是 `getopt` 还是 Boost，你真正理解它们是怎么工作的吗？当遇到奇怪的边界情况时，你能快速定位问题吗？很多时候我们只是"会用"某个库，但并不理解背后的原理。

### 自己动手的真正价值

其实，一个轻量级的命令行参数解析器，核心代码可能也就几百行。但正是这几百行代码，能让我们学到很多有价值的东西：

首先，你会深刻理解命令行参数的解析逻辑。`--output=file`、`-o file`、`-o=file`，这些写法有什么区别？位置参数和命名参数如何共存？必填参数怎么检测？这些问题只有自己实现一遍，才能真正吃透。

其次，这是练习现代 C++ 特性的绝佳场景。我们可以用 `std::string_view` 来避免不必要的字符串拷贝，用模板来实现类型安全的参数获取，用 `std::unordered_map` 来高效存储参数。这些知识点在书本上看过，但真正用起来又是另一回事。

最后，当你拥有完全掌控的代码时，扩展功能就变得随心所欲。想加个子命令？想支持参数别名？想改帮助信息格式？直接改就是了，不用去翻那个可能比你项目还复杂的第三方库文档。

### 我们的起点

在这个教程里，我们从零开始实现一个 ArgParser，它支持：

- 短参数：`-o`、`-v`
- 长参数：`--output`、`--verbose`
- 值参数：`--count=10` 或 `--count 10`
- 布尔开关：`-v`、`--verbose`（出现即为 true）
- 位置参数：按顺序匹配的裸值
- 必填检查：required 参数未传入时报错
- 帮助信息：`--help` 自动生成格式化帮助

代码量不大，但五脏俱全。更重要的是，每行代码都是我们自己写的，每一处设计决策都清清楚楚。

### 环境说明

在开始之前，先确认一下开发环境：

- **编译器**：支持 C++17 的任何现代编译器（GCC 8+、Clang 7+、MSVC 2017+）
- **操作系统**：Linux/macOS/Windows 均可
- **构建工具**：CMake 3.10+（我们会在后面章节详细讲解）

由于我们只使用标准库，不需要安装任何第三方依赖。这使得代码可以轻松移植到任何平台。
